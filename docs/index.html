Архітектура системи

П'ять агентів — один замкнутий цикл

Агенти спілкуються між собою: критик змушує архітектора переосмислити позицію, експерт повертає документ на доопрацювання — доки якість не досягне прохідного балу.

1

Архітектор позову

Аналізує вільний текст ситуації: виявляє сторони, хронологію, правові підстави, позовні вимоги та прогалини у фактах.

IntakeAgent → IntakeResult
↓
2

Фінансовий аналітик

Розраховує судовий збір за ЗСЗ (1% чи 1,5% від ціни позову), перевіряє пільги та визначає правильну підсудність.

FeesCalculator → FeesCalculation
↓
3

Критик / Опонент

Виступає адвокатом диявола: прогнозує заперечення відповідача, виявляє прогалини в доказах, перевіряє строки позовної давності та підсудність. При критичних проблемах — повертає питання до Агента 1.

CriticAgent → CriticReview
↑ Якщо критичні проблеми: питання повертаються до Агента 1 → Агент 2 перераховує збір → Агент 3 перевіряє знову
↓
4

Генератор + Compliance

Генерує повний текст документа з урахуванням зауважень критика. Автоматично перевіряє наявність усіх обов'язкових реквізитів за ст.175 ЦПК / ст.160 КАС / ст.162 ГПК. Формує .docx у форматі судочинства України.

GeneratorV2 + ComplianceChecker → GeneratedDocument
↓
5

Надпотужний Експерт

Партнер з 25-річним досвідом. Оцінює документ за чотирма критеріями (аргументація, відповідність, докази, переконливість). Прохідний бал: 7.5/10 за кожним. Якщо не пройдено — повертає до Агента 4 або Агента 3.

ExpertReviewer → approved / revise_generator / revise_critic
↑ Якщо бал < 7.5: обов'язкові правки → повернення до Агента 4 (переписати) або Агента 3 (переосмислити позицію) → до 3 ітерацій
Технологія

Prompt Caching — радикально дешевші ітерації

Великі статичні блоки (тексти ЦПК, КАС, ГПК, ЗСЗ, стандарти якості) кешуються після першого запиту. Кожна наступна ітерація читає їх за ~10% вартості вхідних токенів.

system_prompt кожного агента
cached_system CACHE HIT → 90% off
Роль агента + тексти кодексів
(ЦПК ст.175-256 + КАС + ГПК + ЗСЗ)
+ стандарти якості
~3 000–8 000 токенів
dynamic_system (не кешується)
Контекст ітерації + зауваження
попередніх агентів
~200–400 токенів
Ітерація 1: cache_write (повна вартість)
Ітерація 2–3: cache_read (~0.1x вартість)
Економія на 3 ітераціях: ~80–90% вхідних токенів
  • 📄

    shared/legal_texts.py

    Центральний файл зі статичними текстами ЗСЗ, ЦПК, КАС, ГПК та стандартами якості (~12 000 символів). Підключається у кожному агенті як кешований блок.

  • ⚡

    ClaudeClient.analyze_cached()

    Новий метод клієнта: розбиває system prompt на cached_system (з cache_control: ephemeral) та dynamic_system. Повертає текст + CacheStats для моніторингу.

  • 📈

    Автоматичний моніторинг

    Після кожного запиту CacheStats логує: cache_creation_tokens, cache_read_tokens, input_tokens, output_tokens — для точного підрахунку витрат.
